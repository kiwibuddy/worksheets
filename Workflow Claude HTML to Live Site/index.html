No prior coding required

If you've been using Claude in your browser, you already know how to talk to it. You ask it to build something, it builds it, you copy the result. Then it sits there. You can't tweak it next week, show it to anyone, or put it on the internet at a real address. Claude Code fixes that. It's the same Claude, sitting on your computer instead of a browser tab, reading your files and saving changes directly.

Before you start
The three tools you'll actually use

If you understand these three roles, the rest of the guide makes sense. You don't need to know how they work, just what each one is there for.

Claude Desktop
Chats with you, reads your files, writes code, edits your project. The brain and the editor.
GitHub
The cloud backup that tracks every version of your project.
Vercel
Turns your GitHub project into a live website at a real URL.
GitHub Desktop
Optional helper for committing and pushing changes if you'd rather click than type.

The flow goes one direction: your brain → Claude Desktop → GitHub → Vercel. You describe what you want, Claude edits the files, your changes get saved to GitHub, and Vercel reads from GitHub and serves your site to the world. Every time you commit a change, your live site updates within about 30 seconds.

Phase 1 · One-time setup
Install everything you need
You only do this once. About 20 minutes the first time. Every project after this skips straight to Phase 2.
Step 1.1
Make sure your Claude account can use Claude Code

Claude Code is included with paid Claude plans (Pro at US$20/month is enough). If you've been using Claude on the free plan, you'll need to upgrade. Go to your Claude account, click your profile, and either confirm you're on Pro or higher, or upgrade.

Step 1.2
Create the two free accounts you'll need

Open two browser tabs. Use the same email address for both so they connect later.

  • github.com, pick a username you're happy being public, it becomes part of every project URL
  • vercel.com, when you sign up, choose "Continue with GitHub". This wires the two together for you.
Why "Continue with GitHub" matters
Signing into Vercel with GitHub is what lets the two talk to each other later. Doing it separately creates a gap you'll have to patch manually.
Step 1.3
Get the Claude desktop app (if you don't have it yet)

There are three likely starting points. Find yours and follow that path.

Path A
I've never used the Claude desktop app
Go to claude.com/download and grab the installer for your operating system (Mac or Windows). Run it like any other app. When it opens, sign in with the same Claude account you use in the browser. Your chat history will appear automatically. Skip to Step 1.4.
Path B
I already use Claude in my browser but never installed the desktop app
Same instructions as Path A. Installing the desktop app doesn't cost anything extra on top of your existing subscription, and it doesn't replace your browser chats. They'll still be there. The desktop version adds the Code tab that this whole guide is built around. Go to claude.com/download and install it.
Path C
I already use the Claude desktop app but only for chatting

You're almost there. You just haven't clicked the right tab yet.

Open the Claude desktop app. Look at the top centre of the window. You'll see tabs, Chat (where you've been living) and Code (where this guide takes you). Click Code.

If clicking Code tells you that you need to upgrade, you're on the free plan. Go to your Claude account, click your profile, and upgrade to Pro (US$20/month). Restart the desktop app. The Code tab will now open.

If clicking Code asks you to sign in online, complete that sign-in in your browser and restart the app. The Code tab will then open to an empty state asking you to choose a project. That's Phase 2 for you. Keep reading.

The Code tab IS Claude Code. Same window, same login, different mode.

Step 1.4
Install GitHub Desktop (the safe-and-simple option)

Download GitHub Desktop and install it. Sign in with the GitHub account you just made. You won't strictly need this, Claude Code can do all the GitHub work through prompts, but having it installed gives you a visual window into what's happening. Reassuring while you're learning, easy to ignore once you're not.

Step 2.1
Get the HTML file out of Claude chat

Open the chat in your browser where Claude built the HTML file for you. There are two ways to get it:

If it appeared as an artifact (the panel that opened to the right of the chat showing a preview): look for the download button in the top-right of the artifact panel. Click it. The file downloads to your usual Downloads folder.

If it was just code in the chat: open a plain text editor (TextEdit on Mac, Notepad on Windows). Copy the entire HTML block from the chat. Paste it into the editor. Save it as index.html.

Step 2.2
Make a project folder on your computer

Decide where your projects will live. Somewhere you'll remember. A folder called Sites or Projects in your home directory works.

Inside that, make a new folder named after this specific project. Use lowercase, no spaces, hyphens between words. For example: family-ai-seminar or spheres-worksheets. This name will show up in URLs later.

Move the HTML file into this new folder. If the file isn't already named index.html, rename it. The name index.html is special. It tells web browsers "this is the home page".

Step 2.3
Open the folder in Claude Code

Open the Claude desktop app. Click the Code tab at the top. You'll see a screen asking you to choose a project, with a button to add a new one. Click it and select the project folder you just made.

Claude Code reads the folder and shows you what's inside. You should see index.html listed. The chat panel on the right is where you'll talk to Claude about this project from now on.

Step 2.4
Run /init to teach Claude about your project

This is the important one. In the chat panel inside Claude Code, type:

/init
A built-in Claude Code command that scans your folder and creates a project briefing

Claude reads your index.html and creates a file called CLAUDE.md, a plain-English briefing about your project. From now on, every new chat in this project starts with Claude already reading it.

Open CLAUDE.md in Claude Code and read it. If anything is wrong or missing, ask Claude to fix it:

Refine the project briefing
Read CLAUDE.md and update it to add these notes: - This is a [describe what your project is, a worksheet or a landing page] - The audience is [who will use this, for example parents at a seminar or students] - Visual style: keep the existing colour palette and font choices. Don't introduce new ones unless I ask. - Always plan a change before writing code. Show me a short plan, wait for me to say "go", then build. - When committing to git, use plain English commit messages (not technical jargon). - Match the voice and tone of any existing copy in the file.
Step 2.5
Add a .gitignore file

One housekeeping file. This tells GitHub "don't ever upload these things": system files, editor caches, anything secret. Type this prompt:

Create the .gitignore
Create a standard .gitignore file for a static HTML project. Include common entries for macOS (.DS_Store), Windows (Thumbs.db), common editors (.vscode, .idea), node_modules in case we add it later, environment files (.env, .env.local), and any common build output folders. Also add CLAUDE.local.md so my personal notes stay off GitHub.

Claude creates the file. You don't need to read it.

3
Phase 3 · Connect to GitHub
Sync your folder with GitHub so it's backed up forever
This is where your work becomes safe. Every version saved, nothing ever lost. About 10 minutes.
Step 3.1
Understand what we're about to do

Right now the project is just a folder on your computer. If the laptop dies, the work is gone. We're about to connect the folder to GitHub so every saved change gets backed up and you can undo anything from yesterday, last week, or last month. The folder doesn't move. GitHub just becomes a mirror.

Step 3.2
The easy way: let GitHub Desktop handle it

Open GitHub Desktop. Choose File → Add Local Repository and point it at your project folder.

GitHub Desktop will tell you the folder isn't yet a Git repository. There's a link in that message saying "create a repository". Click it. Fill in a name (the same as your folder), a one-sentence description, leave the local path as it is, and tick "Initialize with README" if you didn't already make one.

Click Create Repository. GitHub Desktop sets up the connection on your computer.

Now click the big blue Publish Repository button at the top of GitHub Desktop. In the dialog, tick the box that says "Keep this code private", then click Publish Repository.

That's it. Your folder is now synced with GitHub. Open github.com/your-username in a browser and you'll see your new repository at the top of the list.

Step 3.3
The Claude Code way: let Claude do it for you

If you'd rather not click around in another app, ask Claude Code to handle the whole thing. Use this prompt:

Connect this project to GitHub for the first time
I want to put this project on GitHub for the first time as a private repository. Walk me through it step by step. For each step: 1. Tell me what we are about to do and why 2. Give me the exact terminal command and run it for me 3. Wait for me to confirm it worked before moving to the next step Cover: initialising git locally, the first commit, creating the GitHub repo (use the GitHub CLI gh if it's installed, otherwise tell me to create the empty repo manually at github.com/new and then come back), connecting the local folder to it, and pushing the first commit to the main branch. Use my GitHub username, which you can find by running git config user.name if it's set, otherwise ask me.
Always private to start
A public repository means anyone in the world can read your code, including any API keys or notes you've left in files. Always start private.
Step 3.4
Confirm the sync is working

Go to github.com/your-username in your browser and click into the repository. You should see index.html, CLAUDE.md, and .gitignore all listed.

If you see those files, the connection is alive. From this moment on, any change you save and commit will sync to GitHub within seconds.

What "synced" really means
Your folder and GitHub are connected, but not auto-syncing. Changes only go up when you commit them. This is by design. It lets you experiment freely and only push the changes you actually like.
4
Phase 4 · Go live
Deploy to Vercel and get your public URL
From private repo to a live website anyone can visit. About 5 minutes the first time. After this, every change you push deploys automatically.
Step 4.1
Import the repository into Vercel

Go to vercel.com/new. Because you signed in with GitHub, you'll see a list of your repositories. Find the one you just created and click Import.

If your repo doesn't appear, click Adjust GitHub App Permissions and grant Vercel access to it. This is the most common stumbling block.

Step 4.2
Configure the project (for a single HTML file)

Vercel asks a few configuration questions. For a single index.html at the root of your project, the answers are:

  • Framework Preset: Other
  • Build Command: leave blank
  • Output Directory: type a single dot: .
  • Install Command: leave blank

That single dot tells Vercel "the website is right here in the root, no build needed, just serve it". Click Deploy.

Step 4.3
Watch it go live

Vercel takes between 10 and 30 seconds. You'll see a confetti animation when it's done, and a screenshot of your live site. Click the screenshot or the URL underneath. Your project is now live at something like your-project-name.vercel.app.

What just happened
From now on, every commit you push to GitHub triggers a new deployment automatically. You don't have to redeploy. You don't have to upload anything. Save, push, and the live site catches up within 30 seconds.
Step 4.4 · Optional
Connect Vercel directly to Claude Code

This step is optional but worth doing once. It lets Claude see your deployment status, so when a build fails you can ask "why did the deploy fail?" and Claude pulls the logs.

In Claude Code's chat panel, type:

Connect Vercel as an MCP server
Add the official Vercel MCP server to this project. Run this command for me and walk me through authenticating: claude mcp add --transport http vercel https://mcp.vercel.com After it's added, type /mcp and authenticate by following the browser prompt. Then confirm by listing my recent Vercel deployments.

Once connected, you can ask things like "is my site live?", "show me the last 5 deployments", or "the latest build failed, what was the error?", and Claude reads it directly from Vercel without you switching apps.

What is MCP?
Model Context Protocol. It's a way to plug external services (Vercel, GitHub, Gmail, Google Drive, anything) into Claude so Claude can use them on your behalf. You set up each one once. After that, Claude knows how to talk to that service whenever it needs to.
5
Phase 5 · The daily loop
Make changes, save them, see them live
This is the loop you'll run for the rest of the project's life. Once you've done it twice it becomes muscle memory.
Step 5.1
Decide what you want to change

Before you open Claude Code, write down in plain English what you want to be different. One sentence is enough. "Add a pricing section between the hero and the FAQ." Or "Change the call-to-action button to say 'Book a call'."

Vague request, vague result.

Step 5.2
Ask Claude to plan first, then build

This is the most important habit in the entire workflow. Don't ask Claude to "do" something straight away. Ask it to plan first, you confirm the plan, then it builds.

Open the Claude Code chat panel and paste this:

Plan, confirm, then build
I want to make this change to the project: [describe your change in one sentence here] Before you make any code changes, write me a short plan that covers: 1. Which files you'll touch and why 2. What the change will look like (a brief description, not code yet) 3. Anything that might break or that I should decide before we start 4. The roughly correct order of operations Stop after the plan and wait for me to say "go" before writing any code.

Read the plan. Push back on anything that looks wrong. Ask follow-up questions. Once it looks right, type "go" and let it build.

Why this matters
Claude will sometimes propose changes that touch ten files when you only meant to update one button. Catching that in the plan stage takes 30 seconds. Catching it after the AI has rewritten half your project takes an hour and a lot of cleanup.
Step 5.3
Review the changes

Claude Code shows every change as a coloured diff: green for added lines, red for removed. Walk through them. If something looks wrong, click Reject on that specific change. If it looks right, Accept.

The desktop app has a built-in preview pane for HTML files. Open index.html in it, and the preview updates as Claude makes changes. No need to switch to a browser tab.

Checkpoints, your safety net
Claude Code automatically creates checkpoint commits before making changes. If something goes wrong mid-task and you want to wind back to before this conversation started, just press Esc twice. Claude rolls everything back to the last checkpoint. This is your panic button.
Step 5.4
Commit and push

Once you're happy with the changes, save them to GitHub. This makes them live on Vercel within a minute. The simplest way is to ask Claude:

Commit and push
Look at all the changes I've accepted in this session. 1. Write me a clear, short commit message in plain English (not technical jargon). Keep it under 70 characters for the title, then a blank line, then 2 to 3 lines of detail if needed. 2. Show me the message and ask me to confirm or edit it. 3. Once I confirm, run the commit and push to the main branch.

Refresh your live Vercel URL after about 30 seconds. Your change is live.

Or use GitHub Desktop if you prefer clicking
Open GitHub Desktop. You'll see a list of changed files on the left. Type a short commit message at the bottom (e.g. "Add pricing section"), click Commit to main, then click Push origin in the top bar. Same result, no terminal.
Working with data
Adding JSON, CSV, or content files to your project

If you have a JSON of citations, a CSV of stats, or any reference data you want Claude to use, just put the file in the project folder. The sync handles the rest.

Add a data file
The whole process is three actions
  • Drop the file into your project folder using Finder, Explorer, or by dragging it into the GitHub Desktop window
  • Commit and push it (either by asking Claude or via GitHub Desktop)
  • Ask Claude to use it: "read citations.json and add a citation modal to Worksheet 5 using the Pew Research entry"

Claude reads the file directly off your disk. No upload, no re-attaching to a chat, no copy-paste.

Watch out for size limits
GitHub blocks any single file larger than 100 MB. For most worksheet data and citation JSON files you'll be nowhere near this. But if you have a large dataset (5 MB+ of CSV is common), be aware. Anything over 100 MB needs Git LFS or it should live in cloud storage and be referenced by URL. A real database file (.db, .sqlite) should never go in the repo at all. Add the extension to .gitignore instead.
When you're ready
Add your own custom domain

The free your-project.vercel.app URL is fine to share. When you want it on your own domain (like familyaiseminar.co.nz), here's how.

Domain step 1
Buy the domain (or use one you already own)

Two options:

  • Buy through Vercel directly: easiest, no DNS configuration needed, costs around the same as anywhere else. In your Vercel project go to Settings → Domains → Buy.
  • Buy elsewhere and connect it: slightly cheaper sometimes. Use a registrar like Namecheap, Cloudflare, or for NZ domains iwantmyname.
Domain step 2
Connect it in Vercel

In your Vercel project, go to Settings → Domains. Type in your domain. Vercel shows you the DNS records you need to add at your registrar. Usually two records: an A record and a CNAME.

Copy them, log into your registrar, paste them into the DNS settings. Save. Within an hour (often within 5 minutes) Vercel detects the connection and your site loads on your custom domain. SSL certificates are issued automatically and renewed forever. You don't touch anything.

Reference
Plain-English glossary

If a word in this guide didn't make sense, this is where it's defined.

Repository
A folder of files that GitHub is tracking. Often shortened to "repo". Each project has one.
Commit
A snapshot of your project at a moment in time, with a short message saying what changed. Like saving a Word document, but every save is permanent and labelled.
Push
Sending your local commits up to GitHub so they live in the cloud.
Pull
The opposite of push. This brings changes down from GitHub onto your computer. You'll only need this if you edit on multiple devices.
Branch
A parallel version of your project. main is the live version. You can ask Claude to make a new branch (e.g. try-new-design) to experiment without breaking the live site.
Deploy
Publishing your code so people on the internet can see it. Vercel deploys for you automatically every time you push.
CLAUDE.md
The briefing document Claude reads at the start of every conversation about your project. Tells Claude your conventions, audience, and rules.
Checkpoint
An automatic save point Claude Code creates before making changes. Press Esc twice to roll back to one.
MCP
Model Context Protocol, the standard way to plug services like Vercel, GitHub, or Gmail into Claude so Claude can use them on your behalf.
Slash command
A built-in Claude Code shortcut that starts with /. Like /init or /clear.
When things go wrong
Common problems and how to fix them
Problem 1
"Claude made a change I didn't want, but I haven't committed yet"

Press Esc twice. Claude rolls back to the last checkpoint. If you've already accepted some changes you want to keep, instead reject the specific bad ones in the diff view, or ask Claude: "undo only the change you made to the navigation bar, keep everything else."

Problem 2
"I committed and pushed a change that broke the live site"

Use this prompt:

Revert a bad change that's already live
My most recent commit broke something on the live site. I want to undo it but keep all earlier work intact. Show me the last 5 commits with their messages so I can confirm which one to revert. Then create a new commit that safely undoes that one (don't rewrite history) and push it. The live site should roll back automatically once Vercel sees the new commit.
Problem 3
"My change isn't showing up on the live site"

In order, check: did you commit the change? Did you push it? Look at your repo on GitHub in a browser, is the change there? If yes, ask Claude: "check Vercel and tell me the status of my latest deployment, and if it failed, show me the error." (This works best with the Vercel MCP connected from Step 4.4.) If a terminal command failed, paste the error and ask Claude what it means and how to fix it.

Problem 4
"Claude is hallucinating or contradicting itself"

Long chats degrade. After 15 to 20 back-and-forths, Claude starts forgetting what was decided earlier. Type /clear to wipe the chat. CLAUDE.md is reloaded automatically, so the project context survives. Then start your next request with: "read CLAUDE.md and tell me you understand the project before I ask anything."

    A realistic map of the tool
    What can Claude Code actually build?

    The walls are probably further out than you think. Here's the map.

    What Claude Code builds really well

    Anything that lives in the browser

    This is the sweet spot. Websites, landing pages, dashboards, worksheets, presentations, web apps, browser-based tools, interactive documents. You can go from a single index.html (like the one this guide is built around) all the way up to full-scale applications with databases, user logins, payments, and APIs. Claude Code handles the whole spectrum.

    Static sites and content systems

    Blogs, course platforms, documentation sites, single-page portfolios, multi-page brochureware. Anything that renders from files in a folder.

    Backend code and APIs

    If you need a server that handles form submissions, sends emails, processes payments, or stores data, Claude Code can write and deploy it. Standard tools like Supabase, Neon, Firebase, Stripe, and SendGrid are well-supported territory.

    Scripts, automations, and data tools

    Python scripts that process CSVs, scrape websites, generate reports, connect APIs together, or run on a schedule. This is a major strength and wildly underused by non-developers. Quiet automations are often where the compounding gains show up in your second year.

    Native desktop apps for Mac

    Surprising, but real. An independent developer recently shipped a paid, native macOS app to customers where he wrote less than 5% of the 20,000 lines of code himself. The other 95% was Claude Code. This is the frontier, but it works.

    Chrome extensions, VS Code extensions, CLI tools

    All in-scope. Good territory for internal tools you build for yourself or your team.

    What it struggles with or genuinely can't do

    Native mobile apps for iOS and Android

    This is the biggest honest limitation. Claude Code helps a lot with the code itself, but a few things sit between the code and the app store:

    • No App Store account or submission. Apple's developer programme costs US$99/year and requires your signature on legal agreements. Google Play has its own dance. A human has to do that, not Claude.
    • Xcode dependencies for iOS. Building an iOS app requires Xcode running on a Mac. Claude Code can drive Xcode in some scenarios but it's clunky compared to web work.
    • The code isn't always ready out of the box. Mobile app code generated by AI sometimes has accessibility problems or references pieces that don't exist, which can stop the app building. Any generated mobile code needs careful human review.

    There's a solid workaround worth knowing: build it as a web app first, then wrap it using a tool called Capacitor, which turns a web app into an iOS/Android app using the same codebase. This is much more Claude-friendly, though you're still signing up for the App Store compliance dance.

    Legacy or proprietary systems

    Claude Code can struggle with highly customised development environments, proprietary frameworks, or legacy systems that use non-standard patterns. Translation: if you need to integrate with a 15-year-old donor management system that has no documentation online, expect friction. For anything built in the last five years with public documentation, it's fine.

    Visual design from scratch

    Claude Code implements a design beautifully once it has something to reference (a Figma file, an existing page to match, a screenshot, a style guide). But asked to invent a visual identity from zero, results are bland. The fix is simple: give it something to match against. One reference page, one brand palette, one design file. From there, results are strong.

    Things that need a human to sign off

    Buying domains, confirming payment methods, agreeing to terms of service, clicking App Store submission buttons, signing legal documents. Claude flags these and asks you to do them yourself. Not a limitation so much as a boundary, and a healthy one.

    Hardware, printers, plotters, IoT devices

    Not Claude Code's world. It can write code that talks to them, but it can't physically plug them in, calibrate them, or work around driver issues.

    Anything needing true creative judgement at the polish layer

    Naming a product. Deciding which of six legitimate approaches fits your values. Claude can help you think through these, but if you hand them off fully, the output feels like it was designed by a committee.

    Where the ceiling used to be, but isn't anymore

    A few things people still think Claude Code can't do, that it actually can now:

    • Ship a paid product: indie developers have shipped full SaaS products, Mac apps, Chrome extensions, and browser games built primarily by Claude Code.
    • Handle databases and user logins: Supabase integration is mature. A full authentication system is an afternoon's work.
    • Build multi-user apps with real-time features: chat apps, collaborative docs, multi-player games, all feasible.
    • Create AI-powered apps: calling other AI APIs from the app Claude Code is building is its bread and butter.
    • Integrate with payment processors: Stripe integration is standard.
    The honest takeaway
    For nearly everything this workflow is designed for, worksheets, seminars, articles, landing pages, course platforms, ministry sites, internal tools, and automations, you are not going to hit a Claude Code ceiling. Your real limits will be time, budget, and creative direction, not tool capability. The place to hire a human developer is specifically the mobile-app pipeline or a legacy system integration. Everywhere else, you're good.
