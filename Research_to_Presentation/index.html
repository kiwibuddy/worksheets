Before you start
The tools you'll use at each stage

Each tool has one job. They hand off to each other in sequence.

NotebookLM
Reads your sources, connects ideas, and produces a structured research brief.
Claude Chat
Takes the research brief and builds an interactive HTML presentation in an Artifact you can preview and download.
Vercel
Turns your downloaded index.html into a live URL you can share (drag-and-drop, no Git required).

Most presentations start with slides. This one starts with sources. The idea is simple: do the research properly first, get it into a single structured document, and then let Claude build the presentation from that document. The result is an HTML file where your audience can tap any statistic and see exactly where it came from. No more "where did you get that number?"

The flow goes one direction: your sources → NotebookLM → research brief → Claude Chat (Artifact) → HTML file → Vercel → live URL. Each phase produces one output that feeds the next.

Phase 1 · Research
Build a verified research brief in NotebookLM
This is where the quality of your presentation is decided. About 60-90 minutes the first time. Faster on every project after.
Step 1.1
Decide what you're building before you open any tool

Write down three things on paper or in a note:

  • The topic in one sentence. Not "AI" but "How AI is changing the job market for working adults in New Zealand."
  • The audience. Who's in the room? Parents? Students? Church leaders? This changes which stats you pick and how you frame them.
  • The format. A slide-based presentation (for speaking from a stage) or an interactive worksheet (for individual or small-group use)?

Every decision downstream gets easier once these are written down.

Step 1.2
Gather your source materials

Collect 10 to 30 source documents. More is fine (NotebookLM handles up to 50 on the free plan, 300 on Plus). Less than 10 usually means the research is too thin. Good sources include:

  • PDF reports (Pew Research, Barna, WEF Future of Jobs, McKinsey, government reports)
  • News articles (paste the URL directly into NotebookLM)
  • YouTube videos (paste the URL. NotebookLM transcribes them automatically)
  • Google Docs or Slides you've already written
  • Your own notes, outlines, or previous presentation scripts

Save the PDFs to a folder on your computer. You'll upload them in the next step.

Not all sources are equal
A Pew Research report is a primary source. A blog post summarising that report is a secondary source. Build your presentation on primary sources wherever possible. When you cite a secondary source, follow the URL to the original data and confirm the number is accurate. This discipline is the difference between a presentation that holds up to scrutiny and one that doesn't.
Step 1.3
Upload everything to NotebookLM

Go to notebooklm.google.com and create a new notebook. Upload your PDFs, paste your URLs, and add your Google Docs. NotebookLM reads and indexes all of them.

Once your sources are loaded, use the Chat panel to start extracting what you need. Try these prompts:

Extract the strongest evidence
Give me the two most presentation-ready statistics from every source in this notebook. For each statistic, include: - The exact number or percentage - The claim it supports (one sentence) - The source name and publication date - A direct quote if there is one worth using on a slide Format this as a numbered list, grouped by source.

Then follow up with:

Find the narrative thread
Which sources agree with each other? Which ones contradict? Are there any statistics that tell a surprising or counter-intuitive story? Rank the top 10 statistics by how likely they are to change the way an audience thinks about this topic. I want stats that make people stop and reconsider, not stats that confirm what they already believe.
Step 1.4
Generate the Audio Overview (the narrative trick)

This step is optional but it's the single most effective technique in the whole research phase. In NotebookLM's Studio panel, click Audio Overview and let it generate a podcast-style discussion of your sources.

Listen to it if you want. But the real value is the transcript. Download the audio file, then re-upload it as a new source in the same notebook.

Why this works
When NotebookLM builds the Audio Overview, it reads all your sources and synthesises them into one connected narrative. Ideas that were isolated in separate PDFs get linked together. The transcript of that narrative becomes a source itself, giving you a single document that connects your entire research base. It's like having a research assistant write you a briefing paper.
The self-quoting trap
After you upload the transcript, deselect it as a source whenever you ask NotebookLM to generate new Studio items (reports, slide decks, study guides). Otherwise the AI will quote its own summary as if it were a primary source. Use the transcript for narrative structure. Use the original PDFs for facts and figures.
Step 1.5
Export a structured research brief

This is the document that will feed directly into Claude Chat. Use the Chat panel to generate it in one go:

Generate the research brief
Create a research brief I can use to build a presentation. Include these sections: 1. Summary: a one-paragraph overview of the key argument 2. Top 15 statistics: for each one, give me the exact number, the one-sentence claim, the source name, the publication date, a confidence rating (high/medium/low), and the URL 3. Top 5 slide-worthy quotes: with full attribution 4. Narrative arc: a suggested order for presenting these ideas (what comes first, what builds, what's the closer) 5. Full bibliography: every source, with URLs Format this as clean markdown.

Copy the entire output. Open a plain text editor (TextEdit on Mac set to plain text, or Notepad on Windows). Paste it in.

Easiest filename: save as research-brief.txt. Claude reads markdown inside a .txt file exactly the same way — and you avoid macOS silently turning your file into research-brief.md.txt.

If you prefer research-brief.md, that's fine too — just make sure the file really ends in .md.

Mac users (TextEdit)
Before saving, choose Format → Make Plain Text (or TextEdit will save as Rich Text). In the save dialog, uncheck "If no extension is provided, use .txt" so macOS doesn't force an extra .txt on your filename.

This file is the handoff document. Everything Claude needs to build the presentation is now in it.

Phase 2 · Build
Turn the research brief into an interactive HTML presentation
This is where scattered research becomes a polished, clickable presentation. About 45-60 minutes.
Step 2.1
Open Claude Chat and attach the research brief

Go to claude.ai and start a new chat. Drag your research-brief.txt (or research-brief.md) into the message box so it attaches to the conversation.

If you have a screenshot of an existing deck whose look you want to match, attach that image here too and say you want the presentation to echo that style.

What is an Artifact?
When you ask Claude to build HTML, it often opens an Artifact — a live preview on the right side of the screen. You click through your slides there, same as in a browser. There are no project folders, no /init commands, and nothing to install. When you're done, you download a single index.html file.
When Claude Code is still the right tool
If you have a folder of your own images (logos, charts, photos) that must sit next to the HTML on disk, use Claude Code on the desktop app instead — it can work with local files. For a self-contained, text-and-CSS presentation, Claude Chat is the faster path.
Step 2.2
The one-shot build (the big prompt)

Paste the prompt below into the same chat. Your research brief is already attached above.

Build the presentation
Read the attached research brief. Build a single self-contained index.html presentation. Structure: - A launch screen with the title, my name, and a start button - One slide per major idea (aim for 12-18 slides total) - Use the narrative arc from the research brief to order the slides - A closing slide with key takeaways and a QR code placeholder - Slide navigation: keyboard arrows, swipe on mobile, dot indicators Interactive features: - Every statistic should be a clickable card. Clicking opens a modal that shows: the full quote, the source name, the confidence level, and a clickable link to the original source - Speaker notes that toggle with the N key (hidden by default) - A progress bar or slide counter Visual style: - Import Tailwind CSS via CDN for styling so the deck looks polished and modern (similar to a high-end product site) - Use Inter for headings and Fraunces for body text via Google Fonts - 16:9 aspect ratio slides - Dark hero slides for section openers, light slides for content - Animated text entrance (fade-up on slide change) - Include smooth CSS transitions for slide changes and for citation modal open/close - If I attached a reference image for look-and-feel, match its colour palette and typography as closely as you can Data rules: - Use ONLY statistics from the attached research brief. Do not invent numbers. - Every citation modal must include the source URL from the brief - Mark confidence levels: high, medium, or low (from the brief) Plan the slide order first. Show me a numbered list of slide titles with which stats go on each. Stop and wait for me to say "go".
Images and placeholders
Stick to text, typography, and CSS (and inline SVG if helpful). If you ask for photos or stock images, Claude will usually add placeholders — fine for layout, not real pictures. For a single-file deck, that keeps things simple.

Read the plan carefully. Push back on ordering, missing slides, or stats assigned to the wrong slide. Once the plan looks right, type "go".

Step 2.3
Review and refine each slide

The presentation renders in the Artifact panel on the right. Click through every slide there (it behaves like a real web page). For each one, check:

  • Is the headline clear in under 5 seconds?
  • Does the stat card open a modal with the correct source?
  • Is the source URL clickable and correct?
  • Does the speaker note (press N) say what you'd actually say out loud?

Reply in the chat to request changes — be specific:

  • "Make slide 3 a full-bleed dark slide with the stat overlaid in large text"
  • "The modal on slide 7 has the wrong URL. Change it to [correct URL]"
  • "Add a new slide between 4 and 5 about [topic], using this stat from the brief: [quote the stat]"

When you're happy with the draft (you can add the real QR in Step 2.4 after you have a live URL), click the Download icon at the bottom of the Artifact to save index.html to your computer. You'll upload that file in Phase 3.

Step 2.4
Add the QR code and download again

Do this after Phase 3 when you have your real Vercel URL. Stay in the same chat (or start a new chat and re-attach your latest index.html if the session got long).

Generate the QR code
Add a QR code to the closing slide that links to [your-vercel-url]. Generate it as an inline SVG so it doesn't depend on an external service. Make it large enough to scan from a few metres away. Add a small label underneath: "Scan to take this presentation home."

When the Artifact updates, click Download again to save the new index.html. In Phase 3, Step 3.3, you'll drag this updated file back onto Vercel so the live site shows the QR code.

Phase 3 · Deploy
Put your presentation on the internet
From a file on your computer to a live URL anyone can visit. About 5 minutes the first time.
Step 3.1
Create a free Vercel account

Go to vercel.com and sign up. You can use your email — no GitHub account is required for this beginner path.

Pro workflow (optional)
If your organisation already uses GitHub, you can connect a repository to Vercel instead: every git push redeploys automatically. This worksheet uses drag-and-drop so you reach a live URL without touching version control.
Step 3.2
Drag your file to deploy

Open the Vercel dashboard. Click Add New… → Project (wording may vary slightly in the interface). Look for the option to import or upload a static site — then drag the folder that contains your index.html (or drag index.html itself if the uploader accepts a single file) into the upload area.

Vercel deploys static HTML immediately. Within about 30 seconds you get a live URL like your-project-name.vercel.app. Copy that URL — you'll paste it into Claude in Step 2.4.

What just happened
That's it: no Git commands, no repository to configure. Vercel hosts your file and gives you a link you can open on any device. When you change the HTML later, upload the new version to the same Vercel project so your URL stays the same.
Step 3.3
Go back and add the QR code

Now that you have the live URL, go to Step 2.4 in Claude Chat: paste your real URL into the QR code prompt. Download the updated index.html from the Artifact.

Return here and upload or replace that file in your existing Vercel project (same drag-and-drop flow as Step 3.2). After the redeploy finishes, open your URL again — the final slide should show a scannable QR code pointing at your live presentation.

4
Phase 4 · Present and share
Deliver the presentation and let the audience take it home
Everything you need to prepare for the talk and share the result afterwards.
Step 4.1
Test before the event

Open the live URL on the actual device you'll present from. Click through every slide. Tap every stat card and confirm the modal opens with the correct source. Check every URL. Press N to confirm speaker notes appear. Do this the day before the event, not ten minutes before.

Step 4.2
Prepare an offline backup

The HTML file works without internet. Open the index.html file directly from your computer in a browser (double-click it in Finder or Explorer). The slide structure, navigation, modals, and speaker notes all run locally. If the venue has unreliable Wi-Fi, present from the local file instead of the URL.

One caveat: if you used Tailwind via CDN and Google Fonts (as in the build prompt), those load from the internet when you're online. Offline, the deck still works — layout and interactions are fine — but typography may fall back to similar system fonts. For a fully offline-matched backup, ask Claude to inline the critical CSS and font files into the HTML before you download.

Step 4.3
Present from the live URL

During the talk, present directly from the Vercel URL. When someone questions a statistic, tap the stat card on screen. The source modal opens live in front of the audience. This builds credibility faster than anything else you can do as a speaker.

On the final slide, the QR code is visible. Tell the audience: "Scan this and you take the whole presentation with you. Every source, every link, every stat." They walk out with the same interactive file you just presented from.

Step 4.4
Share after the event

Post the URL on social media, email it to attendees, or embed it in a follow-up newsletter. The presentation can keep evolving: edit in Claude Chat, download the new index.html, and drag it onto the same Vercel project to redeploy. The public URL usually stays the same, so people who saved your link see the updated version.

Reference
Plain-English glossary

Terms used in this workflow, defined for non-technical readers.

Research brief
The structured document you export from NotebookLM. Contains your best stats, quotes, sources, and a suggested narrative order.
Citation modal
A popup window that appears when you tap a statistic in the presentation. Shows the source, the full quote, and a link to the original document.
One-shot build
A single prompt that produces the entire first draft of the presentation. You refine from there.
Audio Overview
A NotebookLM feature that generates a podcast-style discussion from your uploaded sources. The transcript is the real output.
Speaker notes
Hidden text on each slide that only the presenter sees. Toggle with the N key during the presentation.
QR code
A scannable barcode on the final slide. Audience members scan it with their phone camera to open the presentation URL.
Primary source
The original research report, dataset, or published study. More reliable than a blog post or news article summarising it.
When things go wrong
Common problems and how to fix them
Problem 1
"NotebookLM is quoting the Audio Overview transcript as a primary source"

You forgot to deselect the transcript before generating new Studio outputs. Go to the Sources panel, untick the transcript source, and regenerate. The transcript is useful for narrative flow. It should never be cited as evidence.

Problem 2
"Claude invented a statistic that isn't in my research brief"

This happens when the prompt doesn't explicitly restrict Claude to the brief. Check every number in the presentation against your saved research brief file. If a stat isn't in the brief, ask Claude to remove it or replace it with one that is. The build prompt includes a "use ONLY statistics from the attached research brief" instruction for this reason.

Problem 3
"The source URL in a citation modal is broken"

The URL was probably wrong in your research brief. Open the brief, find the stat, click the URL yourself. If the original page has moved, search for the report title and update the URL in both the brief and the HTML. Tell Claude: "The URL for the Pew Research stat on slide 5 is wrong. Change it to [correct URL]."

Problem 4
"The presentation looks different on my phone than on my laptop"

The HTML is designed for 16:9 screens (laptop/projector). On a phone, the slides reflow to portrait. This is normal. If a specific slide breaks badly on mobile, tell Claude: "Slide 8 overflows on a narrow screen. Fix the layout so the stat card stacks vertically on mobile."

One-time setup
Connecting the email capture to your Google Sheet

This is a one-off setup. Takes about 5 minutes. After this, every email submitted on this page appears automatically in your spreadsheet.

Setup 1
Create your Google Sheet

Open sheets.google.com and create a new spreadsheet. Call it something like Workflow Email Signups. In row 1, add three column headers: Email, Timestamp, Source.

Setup 2
Create the Apps Script

With the sheet open, click Extensions → Apps Script. Delete any existing code in the editor and paste this:

Google Apps Script — paste this exactly
function doPost(e) { try { var sheet = SpreadsheetApp.getActiveSpreadsheet().getActiveSheet(); var data = JSON.parse(e.postData.contents); sheet.appendRow([data.email, data.timestamp, data.source]); return ContentService .createTextOutput(JSON.stringify({status:'ok'})) .setMimeType(ContentService.MimeType.JSON); } catch(err) { return ContentService .createTextOutput(JSON.stringify({status:'error'})) .setMimeType(ContentService.MimeType.JSON); } } function doGet(e) { return ContentService.createTextOutput('Active.'); }

Click Save (the floppy disk icon or Ctrl+S).

Setup 3
Deploy as a Web App

Click Deploy → New deployment. In the dialog:

  • Type: Web app
  • Execute as: Me
  • Who has access: Anyone

Click Deploy. Google may ask you to authorise permissions — click through and allow. Once deployed, copy the Web App URL it gives you.

Keep this URL private
The Web App URL can receive data from anyone who knows it. Don't post it publicly. It only ever writes to your own sheet.
Setup 4
Paste the URL into the HTML file

Open this HTML file in a text editor (right-click → Open with TextEdit or Notepad). Use Find (⌘ + F on Mac, Ctrl + F on Windows) and search for:

Search for this text in the HTML file:
YOUR_GOOGLE_APPS_SCRIPT_URL

Replace it with your Web App URL. Save the file. Get the updated index.html from Claude Chat (Artifact download), then drag it onto your existing Vercel project to redeploy. Emails will now appear in your spreadsheet within seconds of submission.
