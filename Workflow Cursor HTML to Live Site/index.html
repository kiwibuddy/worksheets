Before you start
The four tools and what each one does

If you understand these four roles, the rest of the guide makes sense.

Claude
Generates the first version of your file. You already have this.
Cursor
Your AI code editor. This is where you make changes from now on.
GitHub
The cloud backup that tracks every version of your file.
Vercel
Turns your GitHub project into a live website at a real URL.

The flow goes one direction: Claude → Cursor → GitHub → Vercel. You build the first draft in Claude, drop it into Cursor to keep editing, push your changes to GitHub for safekeeping, and Vercel reads from GitHub and serves your site to the world. Every time you save a change in Cursor and push it to GitHub, your live site updates within about 30 seconds.

1
Phase 1 · One-time setup
Install everything you need
You only do this once. Set aside 30 minutes the first time. After this, every new project skips straight to Phase 2.
Step 1.1
Create the three free accounts

Open three browser tabs and sign up for each of these. Use the same email address for all three so they connect cleanly later.

  • github.com: pick a username you don't mind being public, this becomes part of every project URL
  • vercel.com: when you sign up, choose "Continue with GitHub". This wires the two together for you.
  • cursor.com: sign in with the same email
Why this matters
Signing into Vercel with GitHub is what lets the two talk to each other later. Doing it separately creates a gap you'll have to patch manually.
Step 1.2
Install Cursor on your computer

Download Cursor from cursor.com and install it like any other app. Open it, sign in with your account, and accept the default settings. The free plan is enough to follow this whole guide.

When Cursor asks you to import settings from VS Code or another editor, just click "Start Fresh" unless you already use VS Code.

Step 1.3
Install GitHub Desktop (the beginner-safe option)

Download GitHub Desktop and install it. Sign in with the GitHub account you just made. This handles the authentication headaches for you, no SSH keys or tokens to manage.

If you prefer the command line
You can use git in Cursor's terminal directly. You'll need to set up either an SSH key or a Personal Access Token to authenticate. GitHub's SSH guide is here. GitHub Desktop skips all that, which is why it's the default recommendation.
Step 1.4
Connect Cursor to GitHub

Open Cursor. On Mac press ⌘ + , for settings (Windows/Linux: Ctrl + ,). Find the section called Accounts or GitHub and click Sign in with GitHub. A browser window opens, click Authorize, and you are done.

This one click means Cursor can handle all your GitHub work from inside the editor with no further setup.

2
Phase 2 · Set up the project
Get your Claude file into Cursor
Take that one HTML file Claude gave you and turn it into a real, version-controlled project. About 10 minutes.
Step 2.1
Make a project folder on your computer

Decide where your projects will live. I keep a single folder called /Sites in my home directory and every project sits inside it. Pick somewhere you'll remember.

Inside that, make a new folder named after this specific project. Use lowercase, no spaces, hyphens between words. For example: family-ai-seminar or spheres-worksheets. This name will show up in URLs later.

Move the HTML file Claude gave you into this folder, then rename it to index.html. The name index.html is special. It tells web browsers "this is the home page".

Step 2.2
Open the folder in Cursor

Open Cursor. Go to File → Open Folder and select the project folder you just made. You'll see index.html appear on the left side. Click it to view the code.

Don't try to read the code. You don't need to understand it, you just need to know it's there.

Step 2.3
Set the ground rules for the AI

Before you let Cursor's AI touch the code, give it a written rulebook. This is a small file that tells the AI how to behave on this project, loaded automatically at the start of every chat. Cursor calls these "rules" and puts them in a folder called .cursor/rules/. You don't need to know how it works, just that creating it now will pay off every time you talk to Cursor after this.

Open Cursor's chat panel with ⌘ + L (Mac) or Ctrl + L (Windows). Make sure Agent mode is selected at the top of the panel. Then paste this prompt:

Prompt 1 · Set the ground rules
Create a folder called .cursor/rules/ in the project root and inside it create a file called main.mdc. The file should have YAML frontmatter with alwaysApply: true and a description. The rules content should specify: - We are working with a single index.html file built originally by Claude - Maintain clean, semantic HTML5 - Use Tailwind CSS via the CDN for any new styling - Follow modern accessibility standards (alt text, aria labels, semantic tags) - Never delete or rewrite large blocks of code without showing me a summary first - Always ask before installing new dependencies - Match the existing visual style and colour palette already in the file

Cursor will create the file. Open it once to confirm it looks sensible. From now on, every conversation you have with Cursor's AI in this project will start with these rules already loaded.

Without rules, the AI fights your conventions on every prompt. With rules, it matches them from the first word.

Step 2.4
Write a brief for the project

Create one more file in the root folder called README.md. This is your project brief, in plain English. It tells future-you (and the AI) what this project is for. Use this prompt:

Prompt 2 · Generate the project brief
Create a README.md file at the project root. Include these sections: 1. What this project is: one or two sentences 2. Who it's for: the audience 3. What it needs to do: three to five bullet points of core functionality 4. What's done: current state of the index.html 5. What's next: the immediate next change I want to make Ask me each question one at a time and wait for my answer before moving to the next. Then write the README based on my answers.

The AI reads this file every time you ask for changes, so time spent here pays off every prompt afterwards.

Step 2.5
Add a .gitignore file

One more housekeeping file. This tells GitHub "don't ever upload these things". Things like macOS system files, editor caches, anything secret. Use this prompt:

Prompt 3 · Create the .gitignore
Create a standard .gitignore file for a static HTML project. Include common entries for macOS (.DS_Store), Windows (Thumbs.db), common editors (.vscode, .idea), node_modules in case we add it later, environment files (.env, .env.local), and any common build output folders.
3
Phase 3 · Backup to the cloud
Push your project to GitHub
This is the moment your work becomes safe. Every version saved, nothing ever lost. About 5 minutes.
Step 3.1
Create the repository

If you installed GitHub Desktop: open it, choose File → Add Local Repository, point it at your project folder. It will say "this folder is not a Git repository", click the link to Create a Repository. Give it the same name as your folder. Choose Private. Click Create, then Publish Repository in the top bar.

If you prefer Cursor's terminal: open it with ⌃ + ` and use this prompt in chat (⌘ + L):

Prompt 4 · First push to GitHub
I want to put this project on GitHub for the first time as a private repository. Walk me through it step by step. For each step: 1. Tell me what we are about to do and why 2. Give me the exact terminal command 3. Wait for me to confirm it worked before moving to the next step Cover: initialising git, the first commit, creating the GitHub repo (use the GitHub CLI if it's installed, otherwise tell me to create it manually at github.com/new), connecting the local folder to it, and pushing the first commit to the main branch. Use my GitHub username, which you can find from git config user.name if it's set.
Choose private, not public
Always start with a private repository. Public means anyone on the internet can read your files, including passwords or sensitive notes you might have left in them. Private is the safe default. You can make it public later.
Step 3.2
Confirm it worked

Go to github.com/your-username in your browser. You should see your new repository at the top. Click in. You should see index.html, README.md, the .cursor folder, and .gitignore.

If you see those files, your project is now backed up to the cloud, version-tracked, and ready to deploy.

4
Phase 4 · Go live
Deploy to Vercel and get your public URL
From private repo to a live website anyone can visit. About 3 minutes the first time. After this, every change you push deploys automatically.
Step 4.1
Import the repository into Vercel

Go to vercel.com/new. Because you signed in with GitHub, you'll see a list of your repositories. Find the one you just created and click Import.

If your repo doesn't appear in the list, click Adjust GitHub App Permissions and grant Vercel access to it. This is the most common stumbling block.

Step 4.2
Configure the project (for a single HTML file)

Vercel will ask you a few configuration questions. For a single index.html at the root, here's what to do:

  • Framework Preset: Other
  • Build Command: leave blank
  • Output Directory: type a single dot: .
  • Install Command: leave blank

That single dot tells Vercel "the website is right here in the root, no build needed, just serve it". Click Deploy.

Step 4.3
Watch it go live

Vercel takes between 10 and 30 seconds. You'll see a confetti animation when it's done, and a screenshot of your live site. Click the screenshot or the URL underneath. Your project is now live at something like your-project-name.vercel.app.

What just happened
From now on, every push to GitHub triggers a new deployment automatically. You don't have to redeploy. You don't have to upload anything. Save, push, and the live site catches up within 30 seconds.
5
Phase 5 · The daily loop
Make changes, save them, see them live
This is the loop you'll run for the rest of the project's life. Once you've done it twice it becomes muscle memory.
Step 5.1
Decide what you want to change

Before you open Cursor, write down in plain English what you want to be different. One sentence is enough. "Add a pricing section between the hero and the FAQ." Or "Change the call-to-action button to say 'Book a call'."

Vague request, vague result.

Step 5.2
Ask the AI to plan first, then build

This is the most important habit in the entire workflow. Don't ask the AI to "do" something straight away. Ask it to plan first, you confirm the plan, then it builds.

In Cursor, open Composer with ⌘ + I (Mac) or Ctrl + I (Windows), make sure Agent mode is on, then paste this:

Prompt 5 · Plan, confirm, then build
I want to make this change to the project: [describe your change in one sentence here] Before you make any code changes, write me a short plan that covers: 1. Which files you'll touch and why 2. What the change will look like (a brief description, not code yet) 3. Anything that might break or that I should decide before we start 4. The roughly correct order of operations Stop after the plan and wait for me to say "go" before writing any code.

Read the plan. Push back on anything that looks wrong. Ask follow-up questions. Once it looks right, type "go" and let it build.

Why this matters
The AI will sometimes propose changes that touch ten files when you only meant to update one button. Catching that in the plan stage takes 30 seconds. Catching it after the AI has rewritten half your project takes an hour and a lot of cleanup.
Step 5.3
Review the changes

Cursor shows every change as a coloured diff: green for added lines, red for removed. Walk through them. If something looks wrong, click Reject on that specific change. If it looks right, Accept.

Open index.html directly in your browser to see what it looks like (right-click the file in Cursor's left panel and choose Reveal in Finder or Reveal in Explorer, then double-click). Refresh the browser tab after each change.

Optional but worth it
Install the Live Server extension from Cursor's Extensions panel. It previews your page in a browser tab that auto-refreshes every time you save.

Before you commit, you can still undo normal editor changes with ⌘ + Z (Mac) or Ctrl + Z (Windows).

Step 5.4
Commit and push

Once you're happy with the changes, save them to GitHub. This makes them live on Vercel within a minute. Use this prompt in Cursor's chat:

Prompt 6 · Commit and push
Look at all the changes I've accepted in this session. 1. Write me a clear, short commit message that describes what changed in plain English (not code-speak). Keep it under 70 characters for the title, then a blank line, then 2 to 3 lines of detail if needed. 2. Show me the message and ask me to confirm or edit it. 3. Once I confirm, give me the terminal commands to stage, commit, and push to the main branch, and run them for me.

Refresh your live Vercel URL after about 30 seconds. Your change is live.

Or use GitHub Desktop
Open GitHub Desktop. You'll see a list of changed files on the left. Type a short commit message at the bottom (e.g. "Add pricing section"), click Commit to main, then click Push origin in the top bar. Same result, no terminal.

Stay simple and keep improving this page

After your first deploy, the simplest path is to keep this as a single-page site and improve it section by section.

Path A · Stay simple
Single-page site

One long scrolling page with anchor sections. Pricing, FAQ, contact, all stacked. A sticky nav at the top scrolls smoothly to each section.

Best for: landing pages, course sales pages, presentation handouts, single worksheets, event pages.

If you chose Path A
Add sections without breaking anything

Use this prompt in Composer (⌘ + I):

Prompt A · Add a new section to a single-page site
I want to add a new [Pricing / FAQ / Testimonials / etc.] section to index.html. Plan it first: 1. Where should it sit in the page order, and why? 2. What existing visual style should it match? (Look at the current colour palette and font choices in the file and reuse them, don't introduce new ones.) 3. What content placeholders do I need to fill in? Then add a sticky top navigation bar (if there isn't already one) with smooth-scroll links to every major section, including the new one. Don't change any existing sections, just add the new one. Stop after the plan. Wait for me to say "go".
When you're ready
Add your own custom domain

The free your-project.vercel.app URL is fine to share. When you want it on your own domain (like familyaiseminar.co.nz), here's how.

Domain step 1
Buy the domain (or use one you already own)

You have two options:

  • Buy through Vercel directly: easiest, no DNS configuration needed, costs around the same as anywhere else. In your Vercel project go to Settings → Domains → Buy.
  • Buy elsewhere and connect it: slightly cheaper sometimes, especially for niche TLDs. Use a registrar like Namecheap, Cloudflare, or for NZ domains iwantmyname.
Domain step 2
Connect it in Vercel

In your Vercel project, go to Settings → Domains. Type in your domain. Vercel shows you the DNS records you need to add at your registrar. Usually two records: an A record and a CNAME.

Copy them, log into your registrar, paste them into the DNS settings. Save. Within an hour (often within 5 minutes) Vercel detects the connection and your site loads on your custom domain. SSL certificates are issued automatically and renewed forever, you don't touch anything.

Reference
Plain-English glossary

If a word in this guide didn't make sense, this is where it's defined.

Repository
A folder of files that GitHub is tracking. Often shortened to "repo". Each project has one.
Commit
A snapshot of your project at a moment in time, with a short message saying what changed. Like saving a Word document, but every save is permanent and labelled.
Push
Sending your local commits up to GitHub so they live in the cloud.
Pull
The opposite of push. This brings changes down from GitHub onto your computer.
Branch
A parallel version of your project. main is the live version. You can make a new branch (e.g. try-new-design) to experiment without breaking the live site.
Deploy
Publishing your code so people on the internet can see it. Vercel deploys for you automatically every time you push.
Terminal
A text window where you type commands directly to your computer. Sounds intimidating, looks intimidating, isn't really. Cursor's AI can write the commands for you.
.cursor/rules/
A folder where you put written instructions for how the AI should behave in this project. Read every time you chat with the AI.
When things go wrong
The four problems you'll hit and how to fix them
Problem 1
"My change isn't showing up on the live site"

In order, check: did you commit the change? Did you push it? Look at your repo on GitHub. Is the change there? If yes, go to your Vercel dashboard, find the project, look at Deployments. There will be a deployment in progress or one that just finished. If it failed, click in to see the error and paste it into Cursor's chat.

Problem 2
"The AI changed something I didn't want it to"

If you haven't committed yet: in Cursor, click Reject on the specific change in the diff view. Or, in the Source Control panel on the left, you can discard all uncommitted changes. If you already committed and pushed: ask the AI to revert that specific commit. Use this prompt:

Prompt · Revert a bad change
My most recent commit broke something. I want to undo it but keep all earlier work intact. Show me the last 5 commits with their messages so I can confirm which one to revert. Then give me the exact terminal command to revert that commit safely (creating a new commit that undoes it, not rewriting history).
Problem 3
"The AI is hallucinating or making things worse"

Long chats degrade. After 15 to 20 back-and-forths, the AI starts forgetting what was decided earlier. Start a fresh chat with ⌘ + N. Begin the new chat with: "Read the README.md and the .cursor/rules/main.mdc, then tell me you understand the project before I ask anything."

Problem 4
"My terminal command failed"

Copy the entire error. Paste it into Cursor's chat and ask what it means. The AI will diagnose it faster than you can google it.

Once you've done it twice
The whole workflow on one screen

Print this. Tape it next to your monitor. After your second project this is all you'll need.

# NEW PROJECT: start to finish
1.Drop Claude's HTML file into a new folder, rename it index.html
2.Open the folder in Cursor
3.Run Prompt 1 (set ground rules) → Prompt 2 (README) → Prompt 3 (.gitignore)
4.Run Prompt 4 (push to GitHub), or use GitHub Desktop's "Publish"
5.Import repo at vercel.com/new. Output Directory: . → Deploy
6.Bookmark the live URL
 
# EVERY CHANGE AFTER THAT
→Open Cursor → ⌘+I → Run Prompt 5 (plan, confirm, build)
→Review diffs, accept or reject
→Run Prompt 6 (commit + push), or use GitHub Desktop
→Wait 30 seconds, refresh live URL

That's it. The first run takes an hour. The tenth project takes 10 minutes.
